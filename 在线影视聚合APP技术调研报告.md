## 一、产品定位修正

## 二、竞品深度分析

### 2.1 TVBox / 影视仓（开源空壳模式）

**产品模式**：开源空壳播放器 + 第三方接口源

| 特性 | 说明 |
|------|------|
| **开源地址** | GitHub开源，可自由修改 |
| **核心机制** | 壳体与视频源分离，通过JSON接口配置数据源 |
| **接口类型** | 爬虫源、XP源、采集源、CMS源 |
| **聚合模式** | 同一影片聚合多个站点资源，可选最优片源 |
| **直播支持** | 支持IPTV/M3U直播源 |
| **平台** | Android/TV/盒子 |
| **特点** | 零成本启动，接口失效风险由第三方承担 |

**接口配置示例**：
```json
{
  "spider": "https://example.com/spider.jar",
  "sites": [
    {
      "key": "douban",
      "name": "豆瓣",
      "type": 1,
      "api": "https://api.douban.com/v2",
      "searchable": 1
    }
  ]
}
```

**代表接口源**：
- 摸鱼接口：`http://我不是.摸鱼儿.com`
- 饭太硬：`http://fty.xxooo.cf/tv`
- 肥猫线路：`http://肥猫.com`
- 巧技：`http://cdn.qiaoji8.com/tvbox.json`

### 3.2 客户端技术选型

| 方案 | 推荐度 | 优势 | 劣势 |
|------|--------|------|------|
| **Flutter** | ⭐⭐⭐⭐⭐ | 一套代码双端，UI一致，性能接近原生 | 视频播放需原生插件 |
| **React Native** | ⭐⭐⭐⭐ | 生态丰富，热更新方便 | 性能略逊于Flutter |
| **原生开发(Kotlin/Swift)** | ⭐⭐⭐ | 性能最佳，体验最原生 | 开发成本高，维护两套代码 |

**推荐：Flutter + 原生视频播放器插件**

**关键插件/库**：
- 视频播放：`video_player` + `chewie`（Flutter封装）或原生 `ExoPlayer`(Android) / `AVPlayer`(iOS)
- 弹幕：`flutter_danmaku` 或自定义Canvas实现
- 网络请求：`dio`
- 状态管理：`Riverpod` / `Bloc`
- 本地存储：`hive` / `sqflite`
- 缓存：`flutter_cache_manager`

### 3.3 后端技术选型

| 方案 | 适用场景 |
|------|---------|
| **Node.js + Express/NestJS** | 快速开发，适合中小型项目 |
| **Java + Spring Boot** | 企业级，高并发，生态成熟 |
| **Go + Gin** | 高性能，适合爬虫调度和高并发 |
| **Python + FastAPI** | 爬虫友好，AI集成方便 |

**推荐：Node.js（快速迭代）或 Go（高性能爬虫）**

### 3.4 数据库选型

| 数据类型 | 推荐数据库 |
|---------|-----------|
| 用户信息/配置 | PostgreSQL / MySQL |
| 影视元数据 | MongoDB（灵活Schema） |

## 四、核心功能模块技术方案

### 4.1 影视资源聚合

#### 方案A：后端集中爬虫（推荐）

```
客户端 → API请求 → 后端爬虫调度 → 并发抓取多个站点 → 聚合去重 → 返回结果
```

**优点**：
- 爬虫逻辑集中管理，易于维护
- 可缓存结果，减少重复请求
- 隐藏真实数据源，降低被封风险

**缺点**：
- 服务器带宽和IP压力大
- 需要处理反爬机制

#### 方案B：客户端直连（TVBox模式）

```
客户端 → 直接请求影视站点API → 本地聚合展示
```

**优点**：
- 服务器压力小
- 用户可自定义数据源

**缺点**：
- 客户端逻辑复杂
- 数据源暴露，易被封
- 难以统一缓存和优化

### 4.2 视频解析技术

| 解析方式 | 原理 | 难度 | 稳定性 |
|---------|------|------|--------|
| **API直链** | 站点提供直链接口，直接获取m3u8/mp4 | 低 | 中 |
| **网页解析** | 解析网页HTML/JS，提取视频地址 | 中 | 低 |
| **JS逆向** | 逆向站点JS加密逻辑，模拟请求 | 高 | 低 |
| **第三方解析接口** | 调用公开解析服务 | 低 | 中低 |

**推荐方案**：
1. 优先对接提供API的CMS站点（如苹果CMS、飞飞CMS）
2. 对无API站点使用Headless浏览器（Puppeteer/Playwright）抓取
3. 建立解析结果缓存机制

### 4.3 弹幕系统

| 实现方式 | 技术要点 |
|---------|---------|
| **实时弹幕** | WebSocket长连接，服务端推送 |
| **弹幕存储** | 按视频ID分片存储，Redis缓存热数据 |
| **弹幕渲染** | 自定义Canvas/Overlay，控制密度和速度 |
| **内容审核** | AI审核接口（百度/腾讯内容安全） |
| **历史弹幕** | 播放时按时间轴加载对应时间段弹幕 |

**Flutter弹幕库选择**：
- `flutter_danmaku`：轻量级，支持基础弹幕类型
- 自定义实现：更灵活，支持高级效果（彩色、定位、表情）

### 4.4 视频播放器

| 功能 | 实现方案 |
|------|---------|
| **基础播放** | 基于ExoPlayer(Android) / AVPlayer(iOS) |
| **多清晰度** | 解析m3u8 master playlist，提供清晰度切换 |
| **倍速播放** | 播放器原生支持 0.5x-3.0x |
| **缓冲策略** | 自定义Buffer配置，预加载下一集 |
| **去广告** | 解析时过滤广告切片（m3u8中识别广告segment） |
| **投屏** | DLNA/AirPlay协议实现 |
| **画中画** | 系统PiP API（Android 8+ / iOS 14+） |

### 4.5 多设备同步

```
设备A播放 → 上报进度到服务端 → 设备B拉取进度 → 续播
```

**同步内容**：
- 播放进度（精确到秒）
- 收藏列表
- 观看历史
- 用户配置（主题、清晰度偏好等）
- 数据源配置

**技术方案**：
- 实时同步：WebSocket
- 离线支持：本地队列 + 网络恢复后批量同步
- 冲突解决：时间戳优先

## 五、数据源方案详解

### 5.1 影视CMS系统对接（最稳定）

国内影视站点多基于以下CMS系统，提供标准化API：

| CMS系统 | API特点 | 示例接口 |
|---------|---------|---------|
| **苹果CMS v10** | RESTful API，文档完善 | `/api.php/provide/vod/` |
| **飞飞CMS** | XML/JSON双格式 | `/index.php/api/vod` |
| **海洋CMS** | 简单HTTP接口 | `/seacms.php` |
| **迅睿CMS** | 现代API设计 | `/api/vod` |

**苹果CMS API示例**：
```http
# 获取分类
GET https://example.com/api.php/provide/vod/?ac=list

# 搜索影片
GET https://example.com/api.php/provide/vod/?ac=detail&wd=影片名

# 获取影片详情
GET https://example.com/api.php/provide/vod/?ac=detail&ids=影片ID
```

### 5.2 爬虫源（灵活但维护成本高）

```python
# 示例：Python爬虫框架
import scrapy

class MovieSpider(scrapy.Spider):
    name = 'movie'
    
    def parse(self, response):
        # 解析影片列表
        movies = response.css('.movie-item')
        for movie in movies:
            yield {
                'title': movie.css('.title::text').get(),
                'url': movie.css('a::attr(href)').get(),
                'cover': movie.css('img::attr(src)').get(),
            }
```

### 5.3 源管理策略

| 策略 | 说明 |
|------|------|
| **内置源** | 应用内置稳定源，开箱即用 |
| **用户自定义** | 支持用户添加私有源（JSON配置/TVBox兼容格式） |
| **源市场** | 服务端维护源列表，用户一键订阅 |
| **源健康检查** | 定时检测源可用性，自动剔除失效源 |

## 七、法律与合规风险

### ⚠️ 重要提醒

| 风险点 | 说明 | 建议 |
|--------|------|------|
| **版权问题** | 聚合未经授权的影视资源涉及侵权 | 仅聚合已授权/公开资源，或提供空壳由用户自行配置 |
| **爬虫合规** | 大规模爬虫可能违反目标站点的ToS | 控制请求频率，遵守robots.txt |
| **数据安全** | 用户数据需符合隐私法规 | 遵循最小化收集原则，提供隐私政策 |
| **内容审核** | 用户弹幕/评论需审核 | 接入AI审核，建立举报机制 |

### 合规建议

1. **空壳模式**：参考TVBox，应用本身不含任何资源，由用户自行配置数据源
2. **免责声明**：明确声明应用仅为播放器，不对第三方内容负责
3. **用户协议**：要求用户遵守当地法律法规
4. **内容过滤**：对已知侵权内容建立黑名单

## 十、总结

### 核心结论

1. **产品模式**：建议采用"空壳+源配置"模式，降低法律风险，参考TVBox生态
2. **技术选型**：Flutter跨平台开发，Node.js/Go后端，PostgreSQL+Redis数据层
3. **数据源**：优先对接苹果CMS等标准化CMS系统，其次开发爬虫
4. **核心差异化**：弹幕系统、多设备同步、去广告、清爽UI
5. **合规优先**：明确免责声明，采用用户自配置源模式

### 下一步建议

1. 确定最终技术栈和团队配置
2. 搭建最小可行原型（MVP）
3. 对接2-3个测试数据源验证技术可行性
4. 设计UI/UX原型
5. 制定详细的开发计划和里程碑
